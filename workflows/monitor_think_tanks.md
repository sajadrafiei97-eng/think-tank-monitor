# Workflow: Arabic Think Tank Monitor

## هدف
بررسی هفتگی وب‌سایت‌های اندیشکده‌های عربی‌زبان و ارسال لینک گزارش‌هایی که حاوی کلیدواژه‌های مشخص (در عنوان یا متن) هستند به بات تلگرام.

---

## ورودی‌ها

| فایل | محتوا |
|------|--------|
| `config.yaml` | لیست اندیشکده‌ها، کلیدواژه‌ها، تنظیمات |
| `.env` | `TELEGRAM_BOT_TOKEN`، `TELEGRAM_CHAT_ID` |
| `.tmp/seen_urls.json` | لینک‌های قبلاً ارسال‌شده (جلوگیری از تکرار) |

---

## مراحل اجرا

```
1. بارگذاری .env و config.yaml
2. بارگذاری seen_urls.json (اگر وجود نداشت، خالی شروع می‌شود)
3. برای هر اندیشکده:
   a. تلاش برای دریافت RSS feed
   b. اگر RSS نداشت یا خالی بود ← scrape HTML
   c. فیلتر لینک‌های قبلاً دیده‌شده
   d. جستجوی کلیدواژه در عنوان/خلاصه
   e. اگر عنوان مطابقت نداشت ← دریافت متن کامل و جستجو
   f. ارسال لینک‌های مطابق به تلگرام
   g. ثبت لینک‌های ارسال‌شده در seen_urls.json
4. نمایش خلاصه نتیجه
```

---

## نحوه اجرا

```bash
# اجرای مستقیم

# اجرا با ذخیره لاگ
run_monitor.bat
```

2. **Create Basic Task** → نام دلخواه
3. **Trigger**: Weekly → روز و ساعت مورد نظر
4. **Action**: Start a Program
   - Program: `C:\Users\javad\Claude\Projects\CSS\run_monitor.bat`
5. **Finish**

لاگ اجرا در `.tmp\run.log` ذخیره می‌شود.

## مدیریت خطاها

| خطا | رفتار سیستم |
|------|------------|
| سایت دسترس‌پذیر نیست | ثبت warning، رد شدن سایت، ادامه |
| خطای Telegram API | ثبت error، لینک در seen ثبت **نمی‌شود** (در اجرای بعدی دوباره تلاش می‌شود) |
| خطای encoding | استفاده از `chardet` برای تشخیص کدگذاری |
| تمام سایت‌ها خطا داشتند | خروج با کد 1 |

## به‌روزرسانی 2026-06-12 — لایه واکشی مستقیم (direct fetch)

### مشکل کشف‌شده
سایت‌های کوچک (مثل `orouba.ps` و `caus.org.lb`) در گوگل ایندکس ضعیف یا با تأخیر دارند؛
جستجوی SerpAPI برایشان «بدون نتیجه» برمی‌گرداند در حالی که مقاله مرتبط روی خود سایت هست.
نمونه واقعی: مقاله «إيران بين الصبر والحسم» (orouba.ps، تاریخ 2026-05-11) — پست 623 همان سایت
(2026-06-04، ایران در متن) هم داخل پنجره ۳۰ روزه بود و گوگل نشانش نمی‌داد.
علاوه بر این، کوتای SerpAPI محدود است و وسط گزارش پوشش rate limit می‌خورد (۱۰ سایت تست‌نشده ماندند).

### راه‌حل
`tools/direct_fetch.py`: مقالات را مستقیم از sitemap سایت (با تاریخ lastmod) یا scrape صفحه اصلی
کشف می‌کند، کلیدواژه‌ها را روی عنوان/slug/متن مقاله تطبیق می‌دهد (با نرمال‌سازی عربی: أإآ→ا و …)
و تاریخ را از متای مقاله (`article:published_time`، `datePublished`) می‌خواند. هیچ کوتای API مصرف نمی‌کند.

* **main.py (مانیتور روزانه):** سایت‌هایی که جستجو هیچ نتیجه‌ای برایشان نداد → direct fetch با همان بازه زمانی.
  مقالات بدون تاریخِ قابل‌تشخیص در حالت روزانه رد می‌شوند تا محتوای قدیمی دوباره ارسال نشود.
* **coverage_report.py (پاس سوم):** سایت‌های بدون نتیجه/خطا/کوتا → چک مستقیم؛ دسته جدید
  «🌐 direct» با نمونه عنوان‌ها. سایت‌های همچنان خالی با «آخرین مطلب مرتبط: تاریخ (N روز پیش، خارج از بازه)»
  حاشیه‌نویسی می‌شوند تا مورد مرزی (مقاله ۳۲ روزه در تست ۳۰ روزه) دیگر شبیه نقطه کور نباشد.

## به‌روزرسانی 2026-06-12 (شب) — معماری تک‌کوئری

### تصمیم کاربر
پس از دو مورد ارسال مقاله خارج از بازه (مقاله ۸ روزه و ۳ ماهه در اجرای «امروز» — هر دو از مسیر
«مقاله‌های بی‌تاریخ» واکشی مستقیم)، کاربر تصمیم گرفت: **جریان روزانه فقط با یک کوئری گوگل از طریق SerpAPI**
و داوریِ تاریخ فقط با عملگرهای `after:`/`before:` خود گوگل.

### جریان روزانه فعلی (`tools/main.py`)
```
(site:تمام‌سایت‌ها OR ...) (intitle:"کلیدواژه‌ها" OR ...) after:Y-M-D before:Y-M-D
```
* یک تماس SerpAPI در هر اجرا (num=50) — حالت «کامل» همین کوئری با کلیدواژه‌های بدون intitle.
* فیلترها: فقط دامنه‌های پیکربندی + حذف doc/xlsx/zip و صفحات index (pdf مجاز است) + یکتاسازی.
* تست پذیرش کاربر (پنجره 2026-06-11→12): هر سه لینک مورد انتظار (مقاله دوحه + نسخه pdf همان + رصانة)
  برگشتند و از فیلترها رد شدند — و هیچ چیز اضافه‌ای نیامد.
* مصرف: ~۱ جستجو per run per سیستم ≈ ماهی ~۶۰ از سهمیه ۲۵۰تایی.
* Google CSE (برای مشتریان جدید بسته شده)، Tavily و واکشی مستقیم **از جریان ارسال حذف شدند**؛
  واکشی مستقیم فقط در `coverage_report.py` برای تشخیص باقی است.

### اصلاح ساعت
ایران از ۲۰۲۲ ساعت تابستانی ندارد (UTC+3:30 ثابت) — کران قبلی 04:30 UTC عملاً **۸ صبح** بود.
اصلاح شد: اجرای روزانه `30 5 * * *` (۹:۰۰ تهران)، مکانیزم پشتیبان ربات هم 05:30.

### خودبازرسی هفتگی
`tools/self_check.py` + `selfcheck.yml` — چهارشنبه‌ها 05:45 UTC (۹:۱۵ تهران): کامپایل ابزارها،
صحت پیکربندی‌ها، سهمیه هر دو کلید سرپ، موفقیت اجرای روزانه اخیر، فعال بودن ربات، سلامت حافظه
لینک‌ها و دسترس‌پذیری سایت‌ها را چک می‌کند و نتیجه را به تلگرام می‌فرستد (گزارش هفتگی ساده قبلی
در monitor.yml حذف شد). اجرای دستی: `python tools/self_check.py --dry-run`

### نکات و محدودیت‌ها
* پنجره زمانی گزارش پوشش لبه تیز دارد: مقاله ۳۲ روزه در تست ۳۰ روزه «بدون نتیجه» حساب می‌شود — حاشیه‌نویسی جدید این را شفاف می‌کند.
* کلیدهای SerpAPI فقط در GitHub Actions Secrets هستند؛ `SERPAPI_KEY` در `.env` محلی خالی است (اجرای محلی main.py بدون کلید خطا می‌دهد).
* وابستگی‌های `fetcher.py` (bs4, lxml, chardet, feedparser) از این تاریخ در `requirements.txt` هستند چون main.py از طریق direct_fetch به آن‌ها نیاز دارد.
* **هر دو سیستم عربی و انگلیسی** را پوشش می‌دهد (کد مشترک است، با هر `--config` کار می‌کند).
  کلیدواژه‌های لاتین با **مرز واژه** تطبیق می‌شوند تا مثلاً iran داخل Tirana نگیرد؛
  کلیدواژه‌های عربی زیررشته‌ای می‌مانند چون پیشوندها به واژه می‌چسبند (وإيران، بطهران).
  تست واقعی 2026-06-12: مقاله ایرانی isis-online.org از صفحه اصلی پیدا شد.
* سایت‌های دارای محافظ ضدربات (مثل stimson.org — خطای 403 برای هر شناسه‌ای) از مسیر واکشی
  مستقیم قابل‌خواندن نیستند؛ پوشش آن‌ها همچنان با موتور جستجوست. این سایت‌ها بزرگ و
  خوب‌ایندکس هستند و مشکل اصلی (ایندکس ضعیف سایت‌های کوچک) شاملشان نمی‌شود.
* مقاله‌های بدون تاریخِ قابل‌استخراج فقط در گزارش پوشش نمایش داده می‌شوند و در پیام‌های
  روزانه ارسال نمی‌شوند (تا محتوای قدیمی به اسم جدید نیاید).
